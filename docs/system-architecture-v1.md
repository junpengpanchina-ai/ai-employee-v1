# AI员工公司 V1 组织架构总纲

## 适用阶段

2–3 个首批 AI 员工起步，目标可平滑扩到 10、30、100+ 员工

## 一、总原则

### 1. 你不直接管理所有 AI 员工

你只管理：

- 老板入口
- AI 总经理
- 关键策略与优先级

也就是：

**老板只面向总经理，不面向所有基层员工。**

### 2. 系统必须可复制、可交接、可授权

这套系统不是给「只有你自己懂」的。它必须满足：

- 新增员工不改底层架构
- 未来助理可接手部分运营
- 未来可接入更多工具和渠道
- 某个员工坏了，不影响整体组织

### 3. 新增员工 = 新增岗位，不是新增一套系统

以后每增加一个 AI 员工，本质上只是：

- 新增一条员工配置
- 新增一个 prompt 策略
- 新增一个输入源定义
- 新增一个输出模板
- 新增一个调度规则

而不是重写 bot、重写数据库、重写主流程。

## 二、组织架构总图

```
老板（你）
   ↓
AI 总经理 / Manager Agent
   ├─ 情报部
   │   ├─ 世界情报员
   │   ├─ 平台动态员
   │   └─ 竞品情报员
   │
   ├─ 机会部
   │   ├─ 赚钱雷达员
   │   ├─ 创业雷达员
   │   └─ 商业模式分析员
   │
   └─ 执行部
       ├─ 报告整理员
       ├─ 数据清洗员
       └─ 工具执行员

人工助理 / 运营协作层
   ↓
Vercel 后台（老板视图 / 助理视图）

底层技术承载：
Telegram = 输入输出
Railway   = 运行与调度
Supabase  = 数据与账本
GRSAI     = 模型算力
Cursor    = 开发与迭代
GitHub    = 版本与协作
```

## 三、各层职责

### A. 老板层（你）

**你负责什么**

- 定方向
- 定优先级
- 看总报告
- 批重点动作
- 决定是否投入跟进
- 判断哪些情报值得升级为项目

**你不负责什么**

- 不直接看底层爬虫日志
- 不直接看每个 worker 状态
- 不直接管理每个 AI 员工细节
- 不直接改数据库
- 不直接碰所有 prompt

**你的理想工作方式**

你每天只需要看到：

- 今日总简报
- 最值得做的 3 个动作
- 哪些重要任务失败了
- 哪些员工表现异常
- 哪些机会值得投入

### B. AI 总经理层（Manager Agent）

这是整个系统的中枢。

**它负责什么**

- 接收你在 Telegram 的命令
- 判断任务属于哪个部门
- 调度下属 AI 员工
- 汇总多个员工结果
- 过滤噪音
- 控制推送频率
- 将重要结果上报给你
- 将低优先级任务交给助理处理

**它不负责什么**

- 不亲自抓所有信息源
- 不亲自做所有执行细节
- 不直接取代基层员工
- 不承担所有领域知识生产

**它的核心价值**

不是「最聪明」，而是：

- 最稳定
- 最会分工
- 最懂组织规则
- 最懂怎么把信息压缩成可决策结果

### C. 部门层

你现在最先做 3 个岗位即可。

#### 1. 情报部

负责「发生了什么」。

**世界情报员**

- 输入：worldmonitor、固定新闻源、官网动态
- 输出：全球 AI / 技术 / 平台情报简报
- 目标：每天给出 3–5 条高价值情报

**平台动态员**

- 输入：OpenAI、Anthropic、Google、Stripe、Vercel、Railway、Supabase 等平台变化
- 输出：平台政策、能力、价格、生态变化
- 目标：提醒你哪些基础设施变化值得跟

**竞品情报员**

- 输入：同类 AI 工具、Telegram 机器人、自动化产品、海外小工具
- 输出：别人怎么做、卖什么、怎么收费
- 目标：帮助你定位自己可以怎么切

#### 2. 机会部

负责「值不值得做」。

**赚钱雷达员**

- 输入：情报部结果 + 商业/产品源
- 输出：可变现机会清单
- 目标：告诉你今天能做什么赚钱动作

**创业雷达员**

- 输入：趋势信息 + 市场缺口 + 你的资源
- 输出：中长期可做产品方向
- 目标：从机会中筛出项目

**商业模式分析员**

- 输入：竞品、收费模式、用户场景
- 输出：订阅、咨询、代运营、工具销售等模式判断
- 目标：避免你只会看热闹，不会设计赚钱结构

#### 3. 执行部

负责「把结果做出来」。

**报告整理员**

把多名员工结果整理成最终日报、周报、专题简报

**数据清洗员**

去重、去噪、提取结构化字段、规范化信息

**工具执行员**

调用 skill、API、抓取工具、文件处理工具

### D. 助理层（未来人工协作层）

这是你现在就必须留孔的一层。

**助理负责什么**

- 看失败任务
- 手动重试
- 调整推送频率
- 启停 AI 员工
- 审核待发送报告
- 标记「高价值 / 低价值」
- 维护非敏感模板
- 协助整理历史简报

**助理不负责什么**

- 不接触密钥
- 不改服务器底层
- 不直改数据库高权限数据
- 不直接操作生产环境命令行
- 不碰核心调度逻辑

## 四、技术系统职责划分

### 1. Telegram

**定位：** 老板入口 + 最终汇报出口

**承担什么**

- 你向 AI 总经理下指令
- AI 总经理向你汇报结果
- 紧急通知
- 主动推送日报 / 突发情报

**不承担什么**

- 不做复杂后台
- 不做历史大数据管理
- 不做配置中心

### 2. Railway

**定位：** 运行层 / 调度层 / 在线办公室

**承担什么**

- bot-service
- orchestrator-service
- worker-service
- cron / 定时任务
- 员工调度
- 调 GRSAI
- 执行基础 jobs

**不承担什么**

- 不做主要数据账本
- 不做复杂管理后台
- 不承担长期业务知识沉淀

**你的理解**

Railway 是：**AI 员工办公楼**

### 3. Supabase

**定位：** 数据层 / 事实账本 / 档案室

**承担什么**

- 员工表
- 部门表
- 任务表
- 消息表
- 报告表
- 反馈表
- 调度表
- 文件索引
- 历史记录

**不承担什么**

- 不直接运行 worker
- 不直接做复杂业务调度
- 不直接接 Telegram

**你的理解**

Supabase 是：**公司账本 + 档案库**

### 4. Vercel

**定位：** 前端层 / 管理后台 / 操作界面

**承担什么**

- 老板视图
- 助理视图
- 员工管理页
- 任务状态页
- 报告查看页
- 审核页
- 配置页

**不承担什么**

- 不做核心常驻 bot
- 不做主要 worker
- 不做核心调度引擎

**你的理解**

Vercel 是：**公司前台 + 管理面板**

### 5. GRSAI

**定位：** 模型算力层

**承担什么**

- 总经理推理
- 情报员总结
- 雷达员判断
- 结构化输出
- 报告归纳

**不承担什么**

- 不做任务存储
- 不做调度
- 不做持久状态管理

**你的理解**

GRSAI 是：**员工的大脑燃料**

### 6. Cursor

**定位：** 开发层 / 原型层 / 提效层

**承担什么**

- 写代码
- 改代码
- 调 prompt
- 重构 skill
- 快速生成 adapter
- 批量改结构
- 本地调试
- 帮你搭出新岗位的代码壳

**不承担什么**

- 不做线上生产运行
- 不做数据库事实源
- 不做真正的员工管理后台

**你的理解**

Cursor 是：**技术研发部的 IDE 和加速器**

### 7. GitHub

**定位：** 代码仓库 + 版本控制 + 协作底座

**承担什么**

- 存放所有代码
- 管理版本
- 发布历史
- 回滚
- 分支协作
- 未来给助理或技术同伴交接

**不承担什么**

- 不做数据库
- 不做线上运行
- 不做员工管理面板

## 六、首批 3 个员工岗位卡

### 1. 世界情报员

**使命**

扫描全球 AI、平台、工具、自动化动态，产出高价值业务情报。

**输入**

- worldmonitor 抓取结果
- 官方博客
- 新闻源
- 产品更新源

**输出**

- 结构化情报简报
- 突发事件提醒
- 对你业务的影响判断

**KPI**

- 每日 3–5 条高价值信息
- 不做新闻搬运
- 每条都附「为什么与你有关」

### 2. 赚钱雷达员

**使命**

从情报中筛选可赚钱、可服务化、可复制的机会。

**输入**

- 世界情报员报告
- 商业模式源
- 海外工具 / 出海产品信息

**输出**

- 今日赚钱机会
- 变现方式
- 起步成本
- 今天可执行动作

**KPI**

- 每天至少 1 条真实可行动机会
- 避免空泛建议

### 3. 创业雷达员

**使命**

从趋势和机会中发现可沉淀成产品、服务、业务线的方向。

**输入**

- 情报部结果
- 赚钱雷达结果
- 你的资源、收款、渠道、模型成本优势

**输出**

- 值得做的 MVP
- 细分市场判断
- 进入路径
- 不值得做的方向

**KPI**

- 每周至少 2–3 条高质量方向判断
- 强调「赛道 + 切入点 + MVP」

## 七、数据库最少要有的表

你后面必须围绕这个账本扩，不要乱长。

### 核心表建议

`employees`、`departments`、`jobs`、`messages`、`reports`、`feedback`、`schedules`、`permissions`、`artifacts`

### 每张表的意义

**employees**

记录每个员工是谁、干什么、状态如何。

**departments**

记录员工归属哪个部门。

**jobs**

记录任务生命周期。

**messages**

记录 Telegram 对话与系统回复。

**reports**

记录结构化输出。

**feedback**

记录你和助理对输出的评价。

**schedules**

记录定时运行规则。

**permissions**

记录员工可调用哪些工具与能力。

**artifacts**

记录文件、报告、外链、产出路径。

## 八、新增员工如何接入而不打乱旧结构

这里是重中之重。

### 新增员工接入 6 步法

**第 1 步：定义岗位卡**

必须先写清楚：

- 岗位名
- 职责
- 输入
- 输出
- 节奏
- 权限
- 失败处理

**第 2 步：注册到 employees**

新增一行，不动旧员工。

**第 3 步：绑定 prompt / template / schedule**

不要把这些硬编码在主流程里。

**第 4 步：绑定部门**

让 AI 总经理知道它属于哪个部门。

**第 5 步：接入权限**

明确它能调哪些工具，不能调哪些工具。

**第 6 步：后台可启停**

助理或你能在 Vercel 后台开关它。

### 原则

**新员工接入不允许：**

- 改主表结构
- 改 Telegram 入口协议
- 改核心调度流程
- 写死在 bot 主代码里

**新员工接入应该：**

- 配置化
- 岗位化
- 可启停
- 可审计
- 可回滚

## 九、Cursor 在整个组织中的正式位置

你特别提到 Cursor，这里单独定一下。

### Cursor 的正式角色

不是生产环境、不是管理后台、不是运行平台。

它是：**研发增幅器**

### Cursor 该怎么用

**场景 1：做新员工原型**

比如要新增「竞品情报员」，你可以让 Cursor：

- 生成岗位代码壳
- 生成 prompt 文件
- 生成 adapter
- 生成 tests

**场景 2：做老员工**

比如调整世界情报员：

- 改输出模板
- 改数据结构
- 改日志逻辑

**场景 3：做工具兼容**

把 GitHub 上的 skill 收编成统一结构。

**场景 4：辅助你和未来技术同伴协作**

通过 GitHub PR / branch 流程来保持版本稳定。

### Cursor 不该直接做的事

- 不直接连生产数据库乱改
- 不直接当线上 bot
- 不直接成为助理操作界面
- 不成为唯一真相来源

## 十、其他潜在系统与建议位置

你说「其他潜在的」，这里直接给你一版。

### 1. GitHub Actions

**定位：** 自动化部署与测试

适合放：

- 测试
- lint
- 自动部署触发
- 生成版本号

### 2. Redis（后续）

**定位：** 队列 / 缓存 / 短期状态

当任务变多后再接入。前期可以不急。

### 3. Sentry / Logtail / 监控工具（后续）

**定位：** 异常监控

当系统开始 7×24 稳定运行时，建议补。

### 4. 对象存储 / Supabase Storage

**定位：** 报告文件、截图、导出结果

适合后续保存结构化产物。

### 5. Notion / 文档系统（可选）

**定位：** 人类知识库

适合给你和助理整理 SOP、业务手册、客户手册。

### 6. OpenClaw / 龙虾（可选内层）

**定位：** 内部技能壳 / 工具调度壳

不再做主入口，只做某些技能执行器。

## 十一、V1 最小落地顺序

### 第一阶段：先活起来

Telegram、Railway、GRSAI、Supabase

**目标：**

- 你发命令
- AI 总经理接收
- 世界情报员产出
- Telegram 回你
- 数据入库

### 第二阶段：再可管理

Vercel 后台、老板视图、助理视图、员工启停、失败重试

### 第三阶段：再可扩张

第二、第三员工、部门划分、权限系统、调度规则、新员工注册流程

## 十二、最后的总定义

这家公司不是：

- 一堆脚本
- 一堆 prompt
- 一堆 bot
- 一堆工具拼盘

这家公司应该是：

- 老板层
- AI 总经理层
- 部门员工层
- 助理协作层
- 数据账本层
- 运行调度层
- 研发提效层

### 最后一条系统宪法

**老板只管理总经理；总经理管理部门与员工；助理通过后台协助运营；新增员工通过岗位制与配置化接入；所有事实统一写入 Supabase；所有在线运行统一由 Railway 承担；Vercel 提供管理与观察界面；Cursor 负责研发与迭代。**
